# Squawk — Architecture

## Goals

- Single deployable service (`collector` + `bot` → `squawk`)
- Stateless: all state lives in the DB, restart anytime without data loss
- Simple: two concerns (data pipeline + digest generation), plain functions, no framework
- Testable: every external dependency behind a Protocol, integration tests against real DB

---

## High-Level Architecture

```
                        ┌──────────────────────────────────────────────┐
                        │                squawk service                │
                        │                                              │
  tar1090 ──────────────► run_pipeline()                               │
                        │   ├─ record sightings (SightingRepository)   │
                        │   └─ enrich_batch()  (EnrichmentRepository)  │
                        │                                              │
                        │ Scheduler ──► generate_digest()              │
                        │ /debug    ──┘      │                         │
                        │                    ├─ DigestQuery             │
                        │                    ├─ DigestRepository        │
                        │                    └─ Broadcaster ──► Telegram│
                        │                                              │
                        │ TelegramBot: /start /stop /debug             │
                        └──────────────────────────────────────────────┘
                                          │
                                     TimescaleDB
```

Two independent concerns:

1. **Pipeline** (`run_pipeline`): continuous loop — polls tar1090, records sightings,
   batches new/expired aircraft, enriches them via Gemini, stores results.
2. **Digest** (`generate_digest`): a function — queries the DB, generates a digest via
   Gemini, caches it, broadcasts via Telegram. Called by the scheduler (weekly cron) or
   the `/debug` command handler.

No event bus, no actor inboxes, no domain events, no replay machinery.

## Table Ownership

**Writes:** each module receives only its own write repository. Cross-table reads
are permitted when explicitly injected.

```
Table                  Owner (write)          Read access
──────────────────────────────────────────────────────────────────────────
aircraft               SightingRepository     DigestQuery
sightings              SightingRepository     DigestQuery
position_updates       SightingRepository     DigestQuery
enriched_aircraft      EnrichmentRepository   DigestQuery, run_pipeline (expiry check)
callsign_routes        EnrichmentRepository   DigestQuery
digests                DigestRepository       —
users                  UserRepository         —
```

`run_pipeline` needs to read `enriched_aircraft.expires_at` to detect TTL expiry.
`EnrichmentRepository.get_expired(hexes, ttl)` exposes this as a read-only query
returning `list[tuple[str, str | None]]` (hex, callsign pairs).

## External Clients

All external HTTP calls go through typed clients in `squawk/clients/`. Each module
defines a Protocol (public interface) and a concrete implementation (private detail).

```python
# squawk/clients/adsbdb.py
@dataclass(frozen=True)
class AircraftInfo:
    registration: str | None
    type: str | None
    operator: str | None
    flag: str | None

class AircraftLookupClient(Protocol):
    async def lookup(self, hex: str) -> AircraftInfo | None: ...

# squawk/clients/routes.py
@dataclass(frozen=True)
class RouteInfo:
    origin_iata: str | None
    origin_icao: str | None
    origin_city: str | None
    origin_country: str | None
    dest_iata: str | None
    dest_icao: str | None
    dest_city: str | None
    dest_country: str | None

class RouteClient(Protocol):
    async def lookup(self, callsign: str) -> RouteInfo | None: ...

# squawk/clients/planespotters.py
@dataclass(frozen=True)
class PhotoInfo:
    url: str
    caption: str

class PhotoClient(Protocol):
    async def lookup(self, hex: str) -> PhotoInfo | None: ...
```

**Retry policy (all clients):** 404 → `None`; 429/5xx → retry with backoff
(up to `max_retries`, default 3); other errors → raise.

## Enrichment

`squawk/enrichment.py` contains the scoring types, AI client, and the batch
enrichment function.

```python
# squawk/enrichment.py

@dataclass(frozen=True)
class ScoreResult:
    score: int           # 1–10
    tags: list[str]
    annotation: str      # one German sentence; empty string if unremarkable

class ScoringClient(Protocol):
    async def score_batch(
        self,
        aircraft: list[tuple[str, AircraftInfo | None, RouteInfo | None]],
    ) -> list[ScoreResult]: ...

async def enrich_batch(
    items: list[tuple[str, str | None]],    # (hex, callsign) pairs
    aircraft_client: AircraftLookupClient,
    route_client: RouteClient,
    scoring_client: ScoringClient,
    enrichment_repo: EnrichmentRepository,
    enrichment_ttl: timedelta,
) -> None:
    """Fetch external data, score via AI, store results.

    1. Fetch AircraftInfo and RouteInfo in parallel for all items.
    2. Call scoring_client.score_batch() — one Gemini call for the whole batch.
    3. Store each result via enrichment_repo.store() (upsert, idempotent).

    Errors: if score_batch fails, logs and returns (batch skipped). Individual
    store failures are logged and skipped — the rest of the batch continues.
    """
```

`_GeminiScoringClient` is the concrete `ScoringClient` implementation, private to
the module. It uses an ADK `LlmAgent` with `output_schema` for structured output.

Key behaviours:
- **Deduplication:** if the same hex appears twice in the batch, it's scored once.
- **Fallback:** if the batch response length doesn't match the input, falls back to
  per-aircraft calls. Per-aircraft failures return `ScoreResult(score=1, tags=[], annotation="")`.

## Digest

`squawk/digest.py` contains the digest types, AI client, and the generation function.

```python
# squawk/digest.py

@dataclass(frozen=True)
class DigestOutput:
    text: str
    photo_url: str | None
    photo_caption: str | None

class DigestClient(Protocol):
    async def generate(
        self,
        candidates: list[dict],
        stats: dict,
        photos: dict[str, PhotoInfo],
    ) -> DigestOutput: ...

async def generate_digest(
    query: DigestQuery,
    digest_repo: DigestRepository,
    photo_client: PhotoClient,
    digest_client: DigestClient,
    broadcaster: Broadcaster,
    period_start: datetime,
    period_end: datetime,
    force: bool = False,
) -> None:
    """Generate and broadcast a digest for the given time window.

    1. Check cache (skipped if force=True). Cache key = (period_end.date(), n_days).
    2. Fetch candidates + stats via DigestQuery.
    3. Fetch photos for top candidates.
    4. Call digest_client.generate() — one Gemini call.
    5. Cache result via digest_repo.
    6. Broadcast to all active users via broadcaster.
    """
```

`_GeminiDigestClient` is the concrete implementation. ADK `LlmAgent` with structured
output, same pattern as scoring.

## Pipeline

`squawk/pipeline.py` — the continuous polling and enrichment loop.

```python
# squawk/pipeline.py

async def run_pipeline(
    poll_url: str,
    poll_interval: float,
    session_timeout: float,
    sightings: SightingRepository,
    enrichment_repo: EnrichmentRepository,
    aircraft_client: AircraftLookupClient,
    route_client: RouteClient,
    scoring_client: ScoringClient,
    enrichment_ttl: timedelta,
    batch_size: int,
    flush_interval: float,
) -> None:
    """Poll tar1090 in a loop, record sightings, enrich new aircraft.

    In-memory state: a `pending` list of (hex, callsign) pairs waiting for
    enrichment. Lost on restart — at most one batch worth of aircraft will
    wait for the next TTL expiry cycle to be re-enriched. Acceptable.

    Crash recovery: close_open_sightings() runs on startup and shutdown. It
    sets ended_at on any sighting with ended_at IS NULL. Safe to call twice
    (no-op when nothing is open).

    Loop body:
    1. Poll tar1090 → list[AircraftState].
    2. Call sightings.record_poll(states, session_timeout):
       - Updates open sightings (last_seen, altitude/distance aggregates).
       - Opens new sightings for unseen aircraft.
       - Closes stale sightings (last_seen older than session_timeout).
       - Returns list[NewSighting] for aircraft new to the aircraft table.
    3. Add new sightings to pending enrichment list.
    4. Check enrichment TTL expiry → add expired (hex, callsign) to pending.
    5. If pending >= batch_size or flush_interval elapsed:
       call enrich_batch(), clear pending.
    6. Sleep poll_interval, repeat.

    Error handling: tar1090 poll failure or record_poll failure → log,
    skip cycle, continue. enrich_batch failure → log, continue (pending
    is cleared regardless — failed aircraft will be retried via TTL expiry).
    """
```

## Bot

### TelegramBot (`squawk/bot/app.py`)

Wraps PTB's low-level API to run under `asyncio.TaskGroup`:

```python
class TelegramBot:
    def __init__(
        self,
        app: Application,
        users: UserRepository,
        on_debug_digest: Callable[[], Coroutine],
        admin_chat_id: int,
    ) -> None: ...

    async def run(self) -> None:
        """
        initialize() → start() → start_polling() → await forever.
        Teardown via try/finally — guaranteed even under TaskGroup cancellation.
        Validated in task 1.2.
        """
```

### Handlers (`squawk/bot/handlers.py`)

```python
def make_handlers(users, on_debug_digest, admin_chat_id):
    # /start — register user
    # /stop  — unregister user
    # /debug — admin only; calls asyncio.create_task(on_debug_digest())
    #          and replies "Digest wird generiert…"
```

`on_debug_digest` is a closure created in `__main__.py` that calls
`generate_digest(period_start=now-24h, period_end=now, force=True)`.
The handler doesn't know about digest internals.

### Broadcaster (`squawk/bot/broadcaster.py`)

```python
class Broadcaster(Protocol):
    async def broadcast(self, digest: DigestOutput) -> None: ...
```

`TelegramBroadcaster` sends to all active users. Handles caption splitting
(Telegram's 1024-char limit). Per-user errors are logged and skipped.

## Main Wiring

`__main__.py` is wiring only. Reading it gives a complete picture of the system.

```python
async def main() -> None:
    config = Config.from_env()
    pool = await create_pool(config.database_url)

    # Repositories
    sightings_repo  = SightingRepository(pool)
    enrichment_repo = EnrichmentRepository(pool)
    digest_repo     = DigestRepository(pool)
    user_repo       = UserRepository(pool)

    # Read query
    digest_query = DigestQuery(pool)

    async with aiohttp.ClientSession() as http:
        # HTTP clients
        aircraft_client = AdsbbClient(http, config.adsbdb_url)
        route_client    = RoutesClient(http, config.routes_url)
        photo_client    = PlanespottersClient(http, config.planespotters_url)

        # AI clients
        scoring_client = _GeminiScoringClient(config.gemini_api_key)
        digest_client  = _GeminiDigestClient(config.gemini_api_key)

        # Telegram
        ptb_app = Application.builder().token(config.bot_token).build()
        broadcaster = TelegramBroadcaster(ptb_app, user_repo)

        # Digest helper — used by scheduler and /debug
        async def _do_digest(period_start, period_end, force=False):
            await generate_digest(
                query=digest_query, digest_repo=digest_repo,
                photo_client=photo_client, digest_client=digest_client,
                broadcaster=broadcaster,
                period_start=period_start, period_end=period_end, force=force,
            )

        async def _debug_digest():
            now = datetime.now(tz=timezone.utc)
            await _do_digest(now - timedelta(hours=24), now, force=True)

        # Scheduler
        async def _scheduled_digest():
            now = datetime.now(tz=timezone.utc)
            await _do_digest(now - timedelta(days=7), now)

        scheduler = APSchedulerBackend()
        scheduler.add_cron_job(_scheduled_digest, config.digest_schedule, tz="Europe/Berlin")
        scheduler.start()

        # Bot
        bot = TelegramBot(ptb_app, user_repo, on_debug_digest=_debug_digest,
                          admin_chat_id=config.admin_chat_id)

        try:
            async with asyncio.TaskGroup() as tg:
                tg.create_task(run_pipeline(
                    poll_url=config.adsb_url,
                    poll_interval=config.poll_interval,
                    session_timeout=config.session_timeout,
                    sightings=sightings_repo,
                    enrichment_repo=enrichment_repo,
                    aircraft_client=aircraft_client,
                    route_client=route_client,
                    scoring_client=scoring_client,
                    enrichment_ttl=config.enrichment_ttl,
                    batch_size=config.enrichment_batch_size,
                    flush_interval=config.enrichment_flush_interval,
                ))
                tg.create_task(bot.run())
        finally:
            scheduler.shutdown()
```

## Migration Plan

Single production instance. Hard cutover — no parallel run.

**Cutover steps:**
1. Stop `collector` and `bot` services
2. Apply new schema (creates new tables, drops event_log if present)
3. Add `last_seen` to `sightings` with backfill (`COALESCE(ended_at, started_at)`)
4. Drop enrichment columns from `aircraft` (moved to `enriched_aircraft`)
5. Truncate `digests` (old format incompatible)
6. Start `squawk` — enrichment backfill happens via TTL expiry on first poll cycles

**What is lost and acceptable:**
- Enrichment scores and annotations (re-generated within hours)
- Cached digest text (regenerated on next Sunday or `/debug`)
- `callsign_routes` (re-fetched during enrichment)
